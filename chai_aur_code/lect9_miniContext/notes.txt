# Context API in React (Complete Explanation)

Since you're learning React, **Context API** is one of the most important concepts to understand. It helps you share data across multiple components without manually passing props at every level.

---

# Why do we need Context API?

Let's say your component structure looks like this:

```text
App
│
├── Navbar
│
├── Home
│     │
│     ├── Products
│     │      │
│     │      └── ProductCard
│     │
│     └── Cart
│
└── Footer
```

Suppose the `App` component has user data:

```jsx
const user = {
  name: "Tanushika",
  age: 20
};
```

Now imagine only the `ProductCard` component needs this data.

Without Context API, you have to pass the data through every intermediate component.

```text
App
 ↓
Home
 ↓
Products
 ↓
ProductCard
```

```jsx
<App user={user} />
```

Then,

```jsx
<Home user={user} />
```

Then,

```jsx
<Products user={user} />
```

Finally,

```jsx
<ProductCard user={user} />
```

Even though `Home` and `Products` don't use `user`, they still have to pass it down.

This is called **Props Drilling**.

---

# The Solution: Context API

Context API allows you to create a **shared data store** that any component can access directly.

Instead of passing props through multiple levels:

```text
App

    UserContext
         │
 ┌───────┼────────┐
 │       │        │
Navbar Home Footer
         │
     ProductCard
```

Now `ProductCard` can directly access the user data from the context.

No props drilling.

---

# The Three Main Parts of Context API

## 1. Create Context

First, create a context.

```jsx
import { createContext } from "react";

const UserContext = createContext();

export default UserContext;
```

Think of this as creating an empty container that will hold shared data.

---

## 2. Provider

The Provider supplies data to all components inside it.

```jsx
<UserContext.Provider value={user}>
    <App />
</UserContext.Provider>
```

The `value` prop contains the data you want to share.

Every child component inside the Provider can access this data.

---

## 3. Consumer (or useContext)

There are two ways to read data.

### Old Way (Consumer)

```jsx
<UserContext.Consumer>
  {(user) => (
      <h1>{user.name}</h1>
  )}
</UserContext.Consumer>
```

This is rarely used today.

---

### Modern Way (useContext)

```jsx
import { useContext } from "react";
import UserContext from "./UserContext";

function Profile() {

    const user = useContext(UserContext);

    return (
        <h1>{user.name}</h1>
    );
}
```

This is the recommended approach.

---

# Complete Example

## Step 1: Create Context

**UserContext.js**

```jsx
import { createContext } from "react";

const UserContext = createContext();

export default UserContext;
```

---

## Step 2: Provide Data

**App.jsx**

```jsx
import UserContext from "./UserContext";
import Profile from "./Profile";

function App() {

    const user = {
        name: "Tanushika",
        city: "Ghaziabad"
    };

    return (
        <UserContext.Provider value={user}>
            <Profile />
        </UserContext.Provider>
    );
}

export default App;
```

---

## Step 3: Consume Data

**Profile.jsx**

```jsx
import { useContext } from "react";
import UserContext from "./UserContext";

function Profile() {

    const user = useContext(UserContext);

    return (
        <>
            <h1>{user.name}</h1>
            <h2>{user.city}</h2>
        </>
    );
}

export default Profile;
```

### Output

```
Tanushika

Ghaziabad
```

# Real-Life Analogy

### Without Context API (Props Drilling)

Imagine a teacher wants to send a message to one student.

```
Teacher
   ↓
Class Representative
   ↓
Monitor
   ↓
Section Leader
   ↓
Student
```

Every person has to pass the message.

---

### With Context API

The teacher simply puts the notice on the notice board.

```
Teacher
    ↓
Notice Board
```

Any student who needs the information can read it directly.

That's exactly how Context API works.

# When Should You Use Context API?

Use it when multiple components need the same data, such as:

* 👤 Logged-in user information
* 🌙 Dark/Light theme
* 🌐 Language settings
* 🛒 Shopping cart
* 🔐 Authentication status
* ⚙️ Global app settings

---

# Context API vs Redux

| Context API                   | Redux                       |
| ----------------------------- | --------------------------- |
| Built into React              | External library            |
| Best for small to medium apps | Best for large applications |
| Easy to learn                 | More advanced               |
| Lightweight                   | Powerful state management   |

> **Context API is React's built-in state-sharing mechanism that allows data to be shared 
across multiple components without manually passing props through every level of the component 
tree (avoiding props drilling). It uses `createContext`, `Provider`, and `useContext` to 
provide global-like access to shared data.**


### One-Line Summary

**Context API lets you share data globally across React components without passing props 
manually through every intermediate component. It solves the problem of props drilling and 
makes state sharing much cleaner.**

# Complete Syntax

```jsx
<UserContext.Provider value={{ user, setUser }}>
    {children}
</UserContext.Provider>
```

Let's break it down.

---

# Part 1: `<UserContext.Provider>`

```jsx
<UserContext.Provider>
```

### What is this?

`Provider` is a special React component that comes from the context you created.

Suppose you created a context like this:

```jsx
import { createContext } from "react";

const UserContext = createContext();
```

Now React automatically gives you two things:

* `UserContext.Provider`
* `UserContext.Consumer`

So,

```jsx
<UserContext.Provider>
```

means:

> "I'm going to provide some data to all the components inside me."

---

# Part 2: `value`

```jsx
value={...}
```

This is the **most important prop** of Provider.

It tells React:

> "This is the data I want to share."

Example:

```jsx
<UserContext.Provider value="Hello">
```

Now every child component can access:

```jsx
const data = useContext(UserContext);
```

Output:

```
Hello
```

---

# Part 3: Why `{ user, setUser }`?

Suppose we have

```jsx
const [user, setUser] = useState(null);
```

Now we want every component to

* read the user
* update the user

So we send both.

```jsx
value={{
    user,
    setUser
}}
```

This is actually an object.

It is the same as writing

```jsx
value={{
    user: user,
    setUser: setUser
}}
```

JavaScript allows the shorter syntax because the property names and variable names are the same.

---

# Visualize it

Suppose

```jsx
const user = {
    name: "Tanushika"
};
```

Then

```jsx
value={{
    user,
    setUser
}}
```

actually becomes

```jsx
value={{
    user: {
        name: "Tanushika"
    },
    setUser: function
}}
```

React stores this object.

---

# Part 4: `{children}`

Suppose in `main.jsx` you wrote

```jsx
<UserContextProvider>
    <App />
</UserContextProvider>
```

Then inside the provider

```jsx
children
```

is

```jsx
<App />
```

So

```jsx
<UserContext.Provider value={{ user, setUser }}>
    {children}
</UserContext.Provider>
```

becomes

```jsx
<UserContext.Provider value={{ user, setUser }}>
    <App />
</UserContext.Provider>
```

---

# What happens now?

Every component inside `App`

```
App
│
├── Navbar
├── Home
├── Login
└── Footer
```

can access

```jsx
const { user, setUser } = useContext(UserContext);
```

without props.

---

# How does `useContext` know what to return?

React looks for the nearest Provider.

```
<UserContext.Provider value={{ user, setUser }}>

      <Navbar />

</UserContext.Provider>
```

When Navbar executes

```jsx
const data = useContext(UserContext);
```

React checks:

> "Which Provider is wrapping me?"

It finds

```jsx
<UserContext.Provider value={{ user, setUser }}>
```

So it simply returns

```jsx
{
    user,
    setUser
}
```

# Why do we wrap `user` and `setUser` inside an object?

Imagine you wrote

```jsx
value={user}
```

Then `useContext` would return only the user.

```jsx
const data = useContext(UserContext);

console.log(data);
```

Output:

```jsx
{
   name: "Tanushika"
}
```

You **cannot update** the user because `setUser` wasn't shared.

---

Now suppose you write

```jsx
value={{ user, setUser }}
```

Then

```jsx
const { user, setUser } = useContext(UserContext);
```

Now you can:

```jsx
console.log(user.name);
```

and also

```jsx
setUser({
    name: "Rahul"
});
```

So you're sharing **both the state and the function to update the state**.

---

## Easy Analogy 📦

Think of `Provider` as a **Wi-Fi router**.

* **`value`** = the internet/data being shared.
* **`children`** = all the devices connected to the router.
* **`useContext()`** = a device asking the router for internet.

```
            Provider (Wi-Fi Router)
        value = { user, setUser }
                   │
     ┌─────────────┼─────────────┐
     │             │             │
   Navbar        Home        Profile
     │             │             │
     └──────── useContext() ─────┘
```

Every child component connected to the Provider can access the shared data without passing it through props. This is exactly why Context API is used.




## Why do we write `value={{ user, setUser }}`?

```jsx
<UserContext.Provider value={{ user, setUser }}>
```

### 1. `value` prop

* `value` is used to **share data** with all child components.
* Whatever is passed inside `value` can be accessed using `useContext()`.

---

### 2. Why `{{ user, setUser }}`?

We pass an **object** because we want to share **multiple values**.

```jsx
value={{
    user,
    setUser
}}
```

is shorthand for

```jsx
value={{
    user: user,
    setUser: setUser
}}
```

* `user` → Current state
* `setUser` → Function to update the state

---

### 3. Why two curly braces?

```jsx
value={{ user, setUser }}
```

* **Outer `{}`** → Enter JavaScript inside JSX.
* **Inner `{}`** → Create a JavaScript object.

---

### 4. Accessing the values

```jsx
const { user, setUser } = useContext(UserContext);
```

Now you can:

```jsx
console.log(user);   // Read the state

setUser(newUser);    // Update the state
```

---

## Quick Interview Answer

> We pass `value={{ user, setUser }}` because `value` can hold any JavaScript value. An object allows us to share multiple related values (the state and its updater function) through a single Context, making them accessible anywhere using `useContext()`.
